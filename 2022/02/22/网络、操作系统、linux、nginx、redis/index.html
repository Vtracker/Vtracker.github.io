    nginx

    一致性hash

    普通hash：取模为当前服务器数，

    一致性hash：取模为2^32 - 1，服务器和ip全部映射，ip顺时针找最近的服务器映射

    改进：虚拟节点，每个服务器映射多个节点（nginx权重为1则为160个）

    redis

    cache aside pattern

    命中：先从缓存中读取数据，命中则直接返回

    失效：先从缓存中读取数据，未命中，从数据库中读取，之后将数据放入缓存

    更新数据：

    1. 数据库（mysql）中修改数据
    2. 将redis中的数据删除

    修改后不更新而是删除数据的主要原因为：防止其他线程更新了之后导致更新顺序不一致（不符合串行结果），写缓存总会出现数据不一致问题

    可引入延迟删除确保该策略一定没问题

    读写分离情况

    若主从同步数据时出现延迟，单机版方案会出现数据不一致，此时可以让redis通过订阅binlog服务来获取最新的数据，为保证高可用防止缓存更新失败，可将binlog日志放入mq中，redis消费数据。如果redis挂了，那么直接放弃更新，查询请求直接查主库。

    缓存击穿

    上分布式锁，只允许一个线程去访问数据库，并更新回缓存中。

    网络

    HTTP

    http 1.0

    1. 简单
    • 明文传输
    • 短连接
    • 无状态

    http 1.1

    1. 引入长连接
    2. 引入cookie
    3. 可以在收到回复前发多个请求（管道网络传输）
    • http请求报文头部未压缩，首部冗长
    • 服务器按顺序处理请求，会发生队头阻塞
    • 没有优先级控制
    • 服务器不能主动推送

    http 2.0

    1. 二进制传输
    2. 头部压缩
    3. 基于https协议，安全
    4. 引入数据流，不同数据流可以并发在一条tcp连接上传输，不同帧有对应的编号
    5. 引入优先级，可以优先处理请求
    6. 服务器能主动推送消息
    7. 多路复用，服务器处理数据可以并发处理请求
    • 仍然采用tcp连接，多个http请求在tcp若堵塞，会触发重传，引起后续请求阻塞

    http 3

    1. 改用udp作为传输层协议，使用QUIC协议确保安全
    2. QUIC协议将原先的tcp三次握手和tls/1.3三次握手优化为三次握手

    https

    请求顺序：

    1. 建立tcp连接，三次握手
    2. 客户端发送hello报文，传输自己支持的tls协议版本和加密算法，以及一个随机数（用于生成会话密钥）
    3. 服务器收到报文，返回选定的协议版本和加密算法，以及数字证书和随机数
    4. 客户端收到，检验CA，用公钥加密本次数据（随机数+摘要）
    5. 客户端和服务器各自有三个随机数，使用协商的加密算法加密生成会话密钥

    http 1.1 优化方案

    1. 减少发送http请求：使用本地缓存url作为key，响应作为value。为防止数据过期，加定时器
    2. 减少http请求次数：多次请求合并为1次请求，
    3. 减少http响应的数据大小：加密

    https 优化方案

    1. 减少tls握手次数：使用tls1.3版本
    2. 硬件优化：使用高级CPU加速计算加密解密
    3. 协议优化：使用ECDHE
    4. 证书优化
