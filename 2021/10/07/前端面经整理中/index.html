HTML5新特性

新增了footer、header等语义化的标签元素，新增了canvas画布、提供了audio和video 的标签、新增了webstroge存储、新增了多线程模块web worker和web socket。

从点击URL到页面生成

首先URL是由域名、端口、文件路径、参数以及可选片段组成，在点击之后，

  1. 根据域名获得ip地址

    域名转化成ip地址首先在浏览器缓存内查找，没有找到则到本地hosts文件查找，仍没有找到到本地网关/路由器，若仍没有则通过DNS服务器，首先传到本地DNS，由本地DNS递归或者迭代查询），得到ip地址

  2. 根据ip地址和端口构成http请求报文（由请求行、请求头部、空行、请求数据构成）的请求头，以及其他数据构成请求头部和数据（POST）

  3. 传输数据

    完整的http请求报文在传输层封装成报文，每个报文加上tcp头部的20字节组装，经由网络层IP协议封装头部进行路由选择，此时协议栈生成ip包和自己的掩码相与，若是在同一网段内部，则1.查询arp缓存，如果有，则单播目的ip对应的mac地址；查不到则广播以太网获得对应的mac地址。如果不在同一个网段，则查询网关mac，单播给网关，网关相与之后发现不在一个网段，则查询路由表进行三层转发。传输过的数据在服务器端层层组装之后得到完整的http请求报文，浏览器根据端口及文件路径获得html文件，向客户端发送http响应报文。

  4. 浏览器根据接收到的响应报文的不同状态码进行相应结果

    浏览器接收到200OK后，根据html文件自上而下构造DOM树，中途遇到js等引入标签会停下进行解析（串行操作），所以一般js标签放在文件最后引入，DOM数构建完后根据css的优先级，行内>内部>外部样式构建css模型树，再与DOM树合并成render tree，在进行布局（确定各个元素的位置和尺寸），之后按render tree进行绘制，若遇到audio和video则并行下载，执行js的时候会先进行同步任务，异步任务例如ajax等在同步任务结束后进行）

http和https

https是http的安全版本，在http的基础上增加了ssl加密。由于http是明文传输，安全性低，加入ssl可以增加安全性，增加中间劫持的成本。

http默认端口80，https默认端口443

https工作过程：

  1. 浏览器发出https请求建立ssl连接
  2. 服务器发送加密的证书文件和网站信息，客户端用公钥解密，比对网站的证书和发来的信息是否一致。
  3. 客户端采用证书加密一段随机值作为密钥，并发送给服务器
  4. 服务器用私钥解密后，得到该随机值，之后用该值进行通话加密

不过https需要CA证书，费用较高，且相比http建立连接的时间更长

TCP三次握手、四次挥手

  1. 客户端应用程序主动打开连接，向服务器发送携带 SYN=1，seq=x的报文段。该报文段不能携带数据（SYN规定不携带数据）。此时客户端处于SYN-SENT
  2. 服务器此时处listen状态（由closed被动打开连接，即应用程序通知服务器需要打开连接），接收到发来的第一段报文，返回一个SYN=1、ACK=1，seq=y，ack=x+1的报文段。该报文段不能携带数据,服务器处于SYN-RCVD
  3. 客户端收到报文，发送ACK=1，seq=x+1，ack=y+1的报文段，客户端处于ESTABLISHED
  4. 服务器收到后，处于ESTABLISHED

三次握手的意义：防止某一次的请求建立连接的报文在中途滞留时间过长，在连接已经建立之后仍发送到了服务器，若是两次握手此时服务器处于建立连接状态而客户端并不会发送数据，造成资源浪费。三次握手时由于客户端收到服务器又一次的SYN报文，并不会回应，此时服务器在经过时延之后自动结束。

四次挥手如图

TIME-WAIT保证自己发送的最后一个报文若是没能送到服务器，服务器能再次发送请求结束报文段

四次挥手比三次握手多的一次在于可以保证服务器能继续发送数据

http请求行

GET：请求指定信息，此时请求报文段一般不包含请求数据

HEAD：类似GET，不过服务器返回的响应报文中没有具体内容，只是为了获得报头（通常为了查看页面状态）

POST：向服务器提交数据，此时请求报文包含请求数据（key-value形式），对传送数据大小没有限制

OPTIONS：允许客户端查看服务器性能

http状态码

状态码在响应报文的信息头（header），为三位数字组成

  • 1xx：指示信息–表示请求已收到，继续处理
  • 2xx：成功–表示请求被成功接受、理解
  • 3xx：重定向–需要进一步操作来完成请求
  • 4xx：客户端错误
  • 5xx：服务器端错误

常见的状态码：

200 OK：客户端请求成功

301 MOVED PERMANENTLY：永久转移，需要重定向，此时会默认以后的ip都为重定向后的ip

302 FOUND：临时移动，类似301，但只是临时使用，客户端仍使用原URL（比如没有登陆）

400 BAD REQUEST：客户端请求错误，不能被服务器理解

403 FORBIDDEN：服务器收到请求，但是拒绝提供服务

404 NOT FOUND：请求资源不存在（比如输入了错误的URL）

500 INTERNAL SERVER ERROR：服务器发生不可预期的错误

cookie、session、sessionStorage、localStorage

cookie是存放在客户端的，用来记录登陆网站的账号名密码、保存上次浏览的页面等信息。cookie始终在同源的http请求中携带，一般大小很小（4kb）。存放在path路径下。cookie的生命周期若没有特殊设定一般在浏览器关闭之后消失，如果在浏览器中设置了cookie的过期时间，则cookie被保存在硬盘中，关闭浏览器仍存在，直到过期时间。并且cookie不是很安全，考虑到安全性应当使用session

session存放在服务器端，服务器通过传过来的cookie中的sessionID来得到session，可以获得该用户的预先设定信息（登录信息购物车信息等等），session的大小没有限制。

sessionStorage存放在客户端，仅在当前会话有效，在同源（协议相同、域名相同、端口相同）的浏览器窗口中，刷新页面或进入同源的另一个页面都不会消失，在关闭了浏览器窗口后销毁。

localStorage存放在客户端，生命周期为永久，只要不主动删除都会一直存在。

webStorage的大小都为5MB，相较于cookie容量更大，且不会传送到服务器 ，本地读取节省了BS交互，速度更快。

TCP和UDP

  1. tcp为面向连接，需要三次握手建立连接，udp为非面向连接，不需要建立连接即可发送数据
  2. tcp提供可靠的服务，拥塞控制，流量控制，有报文序号保证按序接受；udp啥都没有，一直以恒定速率发送，不会重传，需要上层来保证可靠性
  3. tcp的头部为20字节，udp为8字节，头部开销小意味着传输数据效率高
  4. tcp为一对一的单播，udp提供单播、多播、广播
  5. tcp主要应用于对可靠性要求高的情况，比如https、ftp、smtp等；udp适用于大量用户，对质量要求不高但是要求实时性（速度快），比如网络直播、qq视频等

JS垃圾回收机制

js的垃圾回收机制基于“可达”的概念，对于不能从根节点到达的节点，会被标记为不可达，并被引擎回收。

垃圾也就是没有被引用的对象，或者是根访问不到的对象（对象引用环）

具体的执行一般基于标记-清除算法，从根节点开始标记，对其所有子节点（路径为1）的节点进行标记，之后继续BFS，直到最后停下时，没有被标记的节点（对象或者类）会被回收。

对于垃圾回收机制，引擎可以对其进行优化：

  1. 分代回收：对象分为“新对象”和“旧对象”，对于“活得足够久”的对象，会变老，较少接受检查
  2. 增量回收：如果有很多对象，若依次遍历会花费较多时间，所以引擎会将其分为很多小部分，各个部分分别执行，这需要一些额外的标记来跟踪变化，不过延迟会变小，相较于一次大延迟
  3. 空闲时间收集：cpu空闲的时候进行垃圾回收
